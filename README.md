# 超级AI编剧

> **不只帮你“写一个故事”，而是把一个想法推进成有冲突、有选择、可拍、可演、可诊断、可修改的完整剧本。**

“超级AI编剧”是一套面向 Codex 的中文影视剧本 Skill。它可以从一句模糊想法开始，也可以接手人物设定、世界观、大纲或已有剧本，完成故事开发、项目档案、大纲、正式剧本、剧本会诊、台词优化与定稿交付。

适用于短视频、短片、MV、短剧、微电影、长片及系列项目。

## 宣传语

**给我一句话，我还你一部经得起会诊和修改的剧本。**

它不会只堆氛围、套模板或替人物安排巧合。它会持续追问故事真正缺少的东西：人物想要什么，谁在阻止他，失败有什么代价，冲突如何升级，以及人物最终必须做出什么选择。

## 整体流程

```mermaid
flowchart LR
    A[创意或已有文本] --> B[采编与方向开发]
    B --> C[项目档案与大纲]
    C --> D[完整剧本写作]
    D --> E[剧本会诊]
    E --> F[结构与人物修改]
    F --> G[台词优化]
    G --> H[复审与定稿]
    H --> I[Word与对外交付包]
```

### 1. 采编与方向开发

- 从人物、画面、冲突或一句话种子切入
- 根据已有信息决定询问深度，避免重复采访
- 输出本质不同的故事方向，并明确目标、阻力、代价和核心选择
- 在关键节点等待用户确认，不擅自越过创作决策

### 2. 项目档案与大纲

- 建立人物欲望、恐惧、错误信念与关系变化
- 登记世界规则、信息权限、不变清单和待决策事项
- 使用“目标—行动—反作用—代价—状态变化”组织节拍
- 单片、分集项目和复杂世界观分别采用适合的开发粒度

### 3. 正式剧本写作

- 默认使用中文标准影视剧本格式
- 支持国际 `INT./EXT.` 格式与电影化叙事
- 强调可视、可听、可表演，避免不可拍的心理说明
- 每场戏必须改变信息、关系、目标、风险、选择或观众理解
- 附带干净的中文标准剧本 Word 模板

### 4. 剧本会诊

- 从核心前提、主题、人物动机、冲突升级、因果、结构、关系弧线、对白和可表演性九个维度检查
- 问题按 P0、P1、P2 分级
- 每条诊断引用具体场次或原文证据
- 区分根因和表面症状，不在用户未授权时擅自重写

### 5. 修改、复审与定稿

- 支持局部微调、场次重写、结构重排、人物线修复和全稿大改
- 修改后检查因果、伏笔、信息权限、物件、时间、地点和人物状态的联动
- 现实逻辑审核覆盖软件、职业流程、动物行为、空间和时间可行性
- 无未解决的结构性风险后才进入正式定稿

## 主要优点

### 防止故事被“做平”

Skill不会把“治愈、文艺、克制”误解成没有推进。安静场景可以保留，但必须改变关系、信息、选择或观众理解；中段需要产生不同性质的状态变化。

### 不把防平淡等同于强刺激

不同类型使用不同的推进方式：治愈故事关注内部压力与关系变化，喜剧关注误判与后果，悬疑关注信息差与风险，爱情关注关系距离与选择代价。

### 主角必须真正影响故事

巧合可以制造机会，但不能替人物完成高潮。外部事件可以解决客观困境，人物弧线的最终意义必须由主角自己的选择完成。

### 有证据的剧本会诊

不会只说“节奏不够”或“人物不立体”。每个问题都需要对应具体场次、原因、连锁影响和修改原则。

### 支持中文正式剧本交付

内置空白中文标准剧本 Word 模板，包含场景标题、场景描述、角色、人物动作和对白样式，不依赖空格模拟排版。

### 尊重创作确认权

关键阶段设置确认点。Skill不会假装用户已经认可方向、大纲或修改方案，也不会把局部修改扩展成未经授权的全稿重写。

## 安装

仓库中的 Skill 位于：

```text
super-ai-scriptwriter/
```

在 Codex 中使用 Skill Installer，通过以下仓库路径安装：

```text
https://github.com/Doing-Tai-JI/super-ai-scriptwriter/tree/main/super-ai-scriptwriter
```

安装后，在下一轮对话中调用：

```text
$super-ai-scriptwriter
```

示例：

```text
使用 $super-ai-scriptwriter，把“一个凌晨不敢回家的成年人”发展成一部5分钟治愈短片。
```

## 版本规则

本项目采用语义化版本：

- `v1.0.0`：首个稳定公开版本
- `v1.1.0`：新增兼容能力、创作模块或交付功能
- `v1.1.1`：修正规则冲突、格式或小型缺陷
- `v2.0.0`：存在不兼容的流程、结构或行为变化

Git标签和GitHub Release是公开版本的唯一发布依据。开发中的小修改不必立即发布版本；完成一组可验证改进后再升版本。
